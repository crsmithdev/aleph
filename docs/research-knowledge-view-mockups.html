Five ways to keep the graph readable as it grows

The problem isn't the graph — it's the default of showing every node at once. A 300-node force-directed layout is the worst-case view of the data, not the best one. Below are five independent directions for what the Map / Knowledge tabs should look like instead. They can be combined; they can also stand alone.

What breaks today

  • Every concept renders simultaneously, regardless of relevance.
  • Force layout runs per render; 400+ nodes stutters on WSL/low-end hardware.
  • Labels overlap; edges turn into a hairball; finding-count signal is lost.
  • Clicking a node highlights it but doesn't change what else is shown.

What the fix needs to preserve

  • The graph is still the quickest way to spot unexpected connections.
  • Finding-count and cluster structure must remain visible.
  • Deep-links (concept URLs) still need a canonical landing state.
  • Must degrade gracefully: a session with 15 concepts shouldn't feel gated.
01

Spotlight — focus-first by default

Low risk

Land on one concept. Show it plus its 1-hop neighbors. Everything else is a ghost — rendered but de-emphasized, or hidden behind a toggle. Navigate by clicking a neighbor; it becomes the new center. A breadcrumb trail preserves the path so Back never loses context.

Pros

  • Readable at any graph size.
  • Perf scales with visible-subgraph, not total.
  • Matches how people read docs: one thing at a time.

Cons

  • Lose the "wow, look at that structure" overview moment.
  • Cross-cluster jumps need explicit navigation.

Cost

  • Small — current stack.
  • One new component: breadcrumb.
  • ~1 day + interaction polish.
02

Outline + mini-graph — a hybrid reading view

Dual modality

A two-pane view: a hierarchical outline on the left (concepts grouped by cluster, sorted by finding-count), and a small graph on the right showing only the selected concept's neighborhood. Textual thinkers get a scannable list; spatial thinkers still get the structure — but never more of it than they've asked to see.

Neighborhood of —

Pros

  • Keyboard-friendly, screen-reader-friendly.
  • Outline is linkable, sortable, filterable.
  • Graph stays small & legible by construction.

Cons

  • Two panes compete for attention on narrow screens.
  • Hierarchy flattens some cross-cluster links.

Cost

  • Medium
  • Cluster detection (community layer or reuse-parent).
  • ~2–3 days; pairs well with option 1.
03

Semantic zoom — more detail as you lean in

Progressive disclosure

Zoom level controls how many concepts are drawn, not just how big they are. At low zoom, only the highest-centrality concepts are visible with thick edges between them. Zoom in, and the next tier fades in. Zoom more, and you're in a single cluster's detail. Uses the zoom gesture users already know — it just starts meaning something.

DETAIL Level 2 · 14 concepts

Pros

  • Single control for both complexity and zoom.
  • Overview is always the default entry point.
  • Plays well with mouse wheel / pinch.

Cons

  • Tier thresholds need tuning; bad cutoffs hide key concepts.
  • Layout must be stable across tiers — can't re-solve per level.

Cost

  • Medium
  • One-time layout pass; tier-by-centrality buckets.
  • ~2 days; the hardest part is layout stability.
04

Cluster small-multiples — one big graph becomes many small ones

Structural

Run community detection once (Louvain or label-propagation). Render a grid of small cards, one per cluster. Each card shows its internal concepts densely but compactly — and is labeled with what that cluster is about. Cross-cluster links live in a summary strip at the top or appear when a card is expanded. Tufte's small-multiples principle, applied to a knowledge graph.

Pros

  • Cluster structure becomes the first thing you see.
  • Each tile is small enough to be readable.
  • Great for comparing cluster density / health at a glance.

Cons

  • Cross-cluster links need a second visualisation.
  • Auto-clustering quality varies; may need manual overrides.

Cost

  • Large
  • Community detection + per-cluster layout + labeling.
  • ~4–5 days; the clustering quality is the project.
05

3D force-directed — use depth to buy space

Experimental

The overlap problem is fundamentally geometric: too many nodes competing for a 2D plane. Adding a third dimension gives hubs somewhere to go — they pull away from the camera while peripheral nodes ring them. Libraries like react-force-graph-3d (Three.js under the hood) handle thousands of nodes at 60fps. The cost is that 3D is always less scannable than a well-laid-out 2D view: it's an overview mode, not a reading mode.

Lib candidates

react-force-graph-3d — the obvious pick. WebGL, clean API, label sprites.

3d-force-graph — lower-level if more control is needed.

Pros

  • WebGL handles 1000s of nodes cleanly.
  • Depth is a real extra channel for encoding importance.
  • Novel — feels good in a demo.

Cons

  • Occlusion: nodes behind others are literally invisible.
  • Reading a label requires rotation; cognitive cost is real.
  • New dependency (~300 KB); no existing cytoscape styles carry over.

Cost

  • Medium
  • New lib + label sprite pass.
  • ~2 days for a clean v1; polish is open-ended.

Summary

Direction Solves crowding by Best when Cost
01 Spotlight Only rendering the focus + 1-hop. Primary use is reading — inspect a concept, jump to neighbors. Small
02 Outline hybrid Making the list the primary; graph only shows neighborhood. You want keyboard/search-first navigation. Medium
03 Semantic zoom Rendering only the top-N most central at low zoom. Overview → detail is the natural exploration path. Medium
04 Cluster small-multiples Splitting one dense graph into many small ones. Cluster structure is the real story; finding themes matters. Large
05 3D force-directed Buying geometric space with depth. Overview of 500+ concepts; demo-value matters. Medium

Recommendation: ship 01 Spotlight as the new default and layer 02 Outline alongside it as a toggle — these two together fix the reading and navigation problems with one week of work and no new dependencies. Revisit 03 Semantic zoom once finding-count centrality is computed reliably. Treat 04 Clusters as a second-wave project that depends on having working community detection. 05 3D is interesting but belongs later: solve readability first.

All five demos use the same dummy dataset — 32 concepts, 54 edges, 6 clusters — so the relative legibility differences are not because of how the data was shaped for the demo. Open this file in a browser to try the live interactions.
